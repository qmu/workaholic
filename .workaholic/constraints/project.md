# Project

Strategic boundaries for business context, stakeholder priorities, and release management.

## Multi-Plugin Marketplace

**Bounds**: The marketplace contains exactly two plugins: drivin (ticket-driven development) and trippin (AI-oriented exploration). All plugin source lives under `plugins/<name>/`.
**Rationale**: The rename from `core` to `drivin` (v1.0.37) and creation of `trippin` (v1.0.37) established a two-plugin architecture. Documentation, paths, and subagent_type prefixes must reference `drivin` and `trippin`, never `core`.
**Affects**: architecture-manager (component inventory), all leaders (path references in generated docs)
**Criterion**: No active file outside `.workaholic/tickets/archive/` and `.workaholic/stories/` references `plugins/core` or `core:` subagent_type prefix.
**Review trigger**: When a third plugin is added to the marketplace.

## Version Synchronization

**Bounds**: Three version files must always contain the same version string: `.claude-plugin/marketplace.json`, `plugins/drivin/.claude-plugin/plugin.json`, `plugins/trippin/.claude-plugin/plugin.json`.
**Rationale**: The CI validation workflow and marketplace loading mechanism require version parity across all plugin manifests.
**Affects**: delivery-lead (release process documentation)
**Criterion**: All three files contain identical `version` values at every commit on main.
**Review trigger**: When version management automation changes or a new plugin is added.

## Release Cadence

**Bounds**: Releases follow semantic versioning with PATCH increment by default. The `/release` command automates version bumping across all version files.
**Rationale**: PATCH-default prevents accidental breaking changes. Current version is 1.0.38.
**Affects**: delivery-lead (release documentation), recovery-lead (rollback procedures)
**Criterion**: Every release commit message matches `Bump version to v{version}` and all version files are updated atomically.
**Review trigger**: When MINOR or MAJOR version increments become routine.

## Documentation Staleness

**Bounds**: Generated documentation in `.workaholic/specs/` and `.workaholic/policies/` must be regenerated via `/scan` after structural changes (renames, new plugins, new commands).
**Rationale**: Specs and policies are auto-generated by 15 scan agents. Manual patching is error-prone given 316+ stale references across 30 files after the core-to-drivin rename.
**Affects**: architecture-manager (spec generation), quality-manager (policy generation), all leaders (domain-specific docs)
**Criterion**: No spec or policy file contains references to `plugins/core` after a `/scan` run on the current branch.
**Review trigger**: After every `/scan` execution.

## Stakeholder Priority

**Bounds**: The primary stakeholder is the developer (end user). The secondary stakeholder is the plugin author. The tertiary stakeholder is the AI agent (Claude Code). Feature decisions prioritize developer experience over author convenience over agent optimization.
**Rationale**: Derived from stakeholder analysis in stakeholder.md and the human-in-the-loop design principle observed throughout the codebase.
**Affects**: ux-lead (user experience), delivery-lead (release process)
**Criterion**: Any feature that degrades developer approval visibility or adds mandatory autonomous behavior without opt-in violates this constraint.
**Review trigger**: When external contributors join the project or user base grows beyond solo/small-team use.
