# Require a verified deployment-confirmation method in /ship etc

## Summary

`/ship` now requires an established way to confirm a deployment succeeded: it halts and asks for a verification path instead of silently skipping, then executes the confirmation after deploy. Night `/drive` now runs the whole prioritized queue with no per-ticket checkbox.

## Key Changes

- Established the `.workaholic/deployments/` convention (directory rule, frontmatter schema, and a `read-deployments.sh` reader) for deploy procedures and their success-confirmation methods
- Reworked `/ship` from silent-skip into a hard gate: a confirmation method is required, executed after deploy, and a failed confirmation is a failed ship
- Removed the per-ticket checkbox from `/drive` night mode; the invocation authorizes the whole batch, with one group-inclusion question only on mixed topics

## Changes

### Added
- `.workaholic/deployments/` convention and `read-deployments.sh` reader for the deployment contract

### Changed
- `/ship` requires and executes a documented deployment-confirmation method, halting to ask when none exists
- `/drive` night mode runs the whole prioritized queue without a per-ticket checkbox, asking one group question only on distinct topic groups
