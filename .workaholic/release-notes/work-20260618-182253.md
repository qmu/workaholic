# Load the policy lens in workflows and catch doc drift before ship etc

## Summary

The four-pillar engineering policy lens now actually loads into context during workflow commands, `/report` catches stale documentation before a branch ships, and `/trip` is reconciled with current architecture.

## Key Changes

- Injected an always-loaded four-pillar policy index into workflow context, present during `/drive` and the other lensed commands (full policy bodies stay on-demand)
- Added a deterministic documentation-drift detector to `/report`'s release-readiness gate, surfacing stale CLAUDE.md / README.md / SKILL.md before ship
- Reconciled the `/trip` surface with recent architecture: added the policy lens, fixed the stale Trip Ship (merge-last) order, relabeled obsolete plugin-name script homes
- Guarded the committed policy index against silent staleness in CI

## Changes

### Added
- `/report` documentation-drift check: a new `doc-drift.sh` emits structural facts and the release-readiness leaf judges real drift, surfaced as a ship gate and a durable carry-over concern
- Always-loaded four-pillar policy index generated from the pillar `SKILL.md` `## Policies` sections and injected by `policy-lens.sh`; `/drive` brought into the lens

### Changed
- `/trip` surface caught up to recent architecture: policy lens added, Trip Ship flow rewritten to the deploy-confirm-before-merge order, obsolete `core`/`work` script-home labels corrected
- CI Outputs Freshness guard widened to also diff the committed `hooks/policy-index.md`, so a stale index can no longer pass green
