# Write confinement, /request, and a secret rule that judges the value

## Summary

Every workflow's writes are now confined to the current repository, with `/request` as the only sanctioned way to raise work against another one. The release-scan `secret` rule was hardened four times and then restructured: its generic-assignment pass now asks whether the right-hand side *looks like a credential* instead of whether the key name sounds like one â€” turning an ever-growing list of innocent shapes to exclude into a bounded list of guilty ones to match.

## Key Changes

- **Writes stay in this repository.** `rules/general.md` carries the confinement rule (`paths: '**/*'`, so it ships and loads everywhere the plugin is installed), and `hooks/guard-repo-confinement.sh` blocks any `Write`/`Edit` resolving outside the repo and its own worktrees. Worktrees are accepted in both directions, so the mission model keeps working.
- **`/request` is the sanctioned crossing.** It resolves the target repository, reports whether it is public or private, and requires the developer to confirm the exact body verbatim before anything is filed. The confirmation cannot be skipped â€” it is the only control that reaches meaning rather than syntax.
- **The `secret` rule judges values, not names.** Pass 2 flags a right-hand side only when it is a quoted string starting alphanumeric, or a bare run of value characters ending the line. Identifiers, dotted paths, **calls** (`apiKey: keyOption()`), templates, environment reads, type annotations and `Token::Path` are references and are no longer enumerated at all â€” six subtractions were deleted, and five of them needed no replacement.
- **Credential detection got stricter, not looser.** Keywords carrying a suffix (`SECRET_KEY`, `aws_secret_access_key`, `access_key_id`, `refresh_token_value`) are detected â€” including Django's `SECRET_KEY` and the exact key name AWS's own config files use.
- **An artifact can advance several missions.** The `mission:` relation is many-valued (`mission: [alpha, beta]`), read through one shared reader, and every commit seam rolls all of them. `/report` no longer asks a developer to pick one and silently drop the other.

## Changes

### Added
- Repository write-confinement: the rule in `rules/general.md` plus the blocking `PreToolUse(Write|Edit)` guard `hooks/guard-repo-confinement.sh`.
- `/request` â€” the only sanctioned path for filing a ticket in another repository, with target visibility reporting and a non-skippable verbatim-body confirmation.
- A many-valued `mission:` relation, with `mission/scripts/read-relation.sh` as the single reader.

### Changed
- The `secret` rule's generic-assignment pass now matches on the value rather than the key name. Behaviour for real credentials is unchanged; ordinary code that merely *references* a secret is no longer reported on the one tier that cannot be waived.
- The mission lens stays silent inside a worktree that owns no mission, and for a mission with no acceptance criteria (`0/0` is a technical condition with nothing to act on). `/mission summary` deliberately keeps the lower bar.
- `/explain` stages its intermediate HTML inside the repository; the exported PDF still lands in your chosen directory.

### Fixed
- Credential keywords carrying a suffix are no longer missed.
- TypeScript type annotations (`let nextToken: string | undefined;`) and scope resolution (`Token::Path`) no longer hard-block `/ship`.
- A false negative introduced while suppressing those annotations â€” a bare word after a key parsed as a type, so `password: mysecret123` stopped being flagged â€” is repaired.

### Removed
- The `leak` rule's internal-hostname pattern. It detected zero of five real leaks while misfiring on `metadata.internal`, a field every script-bearing skill is required to carry. The documents claiming the scan machine-enforces the keep-context-generic convention were corrected: it does not, and cannot.
