# Add /monitor parallel mission driver etc

## Summary

Triaged the deferred-concern corpus to zero and hardened the workflow machinery it protects, then built `/monitor` — a parallel-missions executor — on that floor. The interrogation and monitor contracts were each corrected from first use.

## Key Changes

- Added `/monitor`: a developer-confirmed pre-flight, then one autonomous drive per mission worktree in parallel, looping until every mission completes or only explicitly deferred escalations remain — validated live with two concurrent missions
- Refined `/monitor` on first-use feedback: blockers are pushed as one-at-a-time decisions, the orchestrator is a non-blocking dispatcher, and dev environments boot per worktree at dispatch
- Drove seven themed hardening tickets from the concern triage (commit.sh gates, catch scanner, release-scan tiering, concern machinery, mission floors, worktree guards)
- Sharpened the quality-gate interrogation to ask only genuine developer decisions and draft derivable criteria into the ticket
