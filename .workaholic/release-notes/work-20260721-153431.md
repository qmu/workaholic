# Harden the mission and concern lifecycles etc

## Summary

The mission and concern lifecycles get safety guards, developer-centric views, and prevention-first management. Worktree cleanup no longer risks the default branch, bare `/mission` becomes a planning session, and the concern corpus rebalances from disposal to prevention.

## Key Changes

- Worktree teardown deletes only ephemeral `work-*` branches; `main` and hand-made branches are preserved
- Bare `/mission` is now a developer readiness session: status → replan to drive-ready → strategy-gap discussion → `/goal /monitor ok` hand-off
- Lane-aware concern triage: the prompt fires on your own mission lane, not the global total
- Concern promotion floor: the story keeps every concern; only moderate+ (or explicitly kept) enter the tracked corpus
- Reversible, developer-confirmed demotion pass to shrink an already-bloated corpus, plus OKF type validators for stories and trips

## Changes

### Added

- `validate-story.sh` / `validate-trip.sh` — OKF `type` validators for new story/trip writes, grandfathering git-tracked history
- Concern promotion floor in extraction (`CONCERN_PROMOTE_MIN`, default `moderate`) and a developer-confirmed demotion pass (`propose-demotions.sh` / `demote-concern.sh`, `status: demoted`)
- Computed `ready`/`ready_reason` on missions and `active_missions` on strategies, driving the `/mission` planning session and its strategy-gap discussion

### Changed

- Worktree cleanup guards the default branch (deletes only `work-YYYYMMDD-HHMMSS`)
- Bare `/mission` centers on the caller (two-tier view from computed `relation`) and runs a readiness session; the `summary` mode is retired into it
- Concern triage is lane-aware (concerns carry an `owner`; `should_triage` is lane-scoped)
