# Enforce the working-directory and layout guards by default etc

## Summary

The working-directory guard and the `.workaholic/` layout gate now block by construction whenever the plugin is installed — no environment-variable or marker opt-out that fails open off the happy path. A verifiable closed-layout / lockstep-registration policy plus a CI merge gate keep the layout's two sources of truth from drifting.

## Key Changes

- Both guards block **unconditionally** — enforcement is built into the plugin code, so "plugin installed = guard active"; `WORKAHOLIC_ENFORCE_CWD`, `WORKAHOLIC_STRICT_LAYOUT`, and the `.strict-layout` marker are removed
- The layout allowlist is reconciled against a real `layout-doctor` run (`strategies`, `guides`, `policies`, `scan-allow`, `leak-denylist`), so legitimate writes still pass
- A verifiable CLAUDE.md closed-layout / lockstep-registration policy: a new `.workaholic/` artifact directory must update both the allowlist and the `rules/workaholic.md` table in the same change
- `layout-doctor.sh` is wired into Local Verification and a `Validate Plugins` CI gate that fails the merge on a drifted allowlist

## Changes

### Added
- CLAUDE.md closed-layout / lockstep-registration policy, backed by a `Validate Plugins` CI audit gate and a Local Verification command

### Changed
- Working-directory guard now denies a top-level cwd-moving `cd` unconditionally; the `.workaholic/` layout gate now blocks writes into undesignated subdirectories unconditionally — both env-var/marker opt-outs removed and the retired two-mode design rewritten out of all docs
