## 7. Successful Development Patterns

- **Triage before fixing.** Deduplicating the corpus first (21 → 7) revealed the true work — 4 actionable issues, not 21 — and prevented redundant effort on chained duplicates.
- **Verify, then document instead of duplicating.** The build.mjs orphan concern turned out to be already guaranteed by the wholesale `rmSync`; reading the code first avoided shipping redundant cleanup logic and produced a guarding comment instead.
- **One pattern definition, shared by build and lint.** Extracting `script-ref-patterns.mjs` so `build.mjs` and `verify.mjs` agree on exactly one accepted cross-skill reference form prevents a reference passing the lint yet missed by the build.
- **Advisory checks over hard blocks.** The `/ship` capability check warns and steers without overriding the authoritative confirmation gate — useful guidance that never wedges a ship.
- **Disposition records preserve memory.** Closing accepted/deferred concerns with a recorded disposition (rather than deleting) keeps the audit trail while clearing the active rotation.
