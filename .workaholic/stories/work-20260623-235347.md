## 1. Overview

This branch unifies `/trip` and `/drive` on the **ticket** as their single unit of work. It adds a trip Planning-phase **Decomposition gate** (design → tickets), reworks the trip Coding phase into a **per-ticket drive** with three-agent QA, converges `/report` so trips report as richly as drives, and makes `/trip` **context-aware** so a populated todo queue is trip-driven directly (the `ticket → trip` direction) — with the whole "sources × executors" model documented.

**Highlights:**

1. Trip Planning phase emits tickets from the agreed design (Decomposition gate), each carrying `## Policies` and a Trip Origin link
2. Trip Coding phase becomes a per-ticket drive loop with three-agent QA as the gate (Constructor → Architect → Planner, no developer prompt)
3. `/report` trip-mode builds Changes from archived tickets and links the trip rationale
4. Context-aware `/trip`: a populated todo queue runs queue-execute (skip design, trip-drive the tickets)
5. The unified sources × executors model documented in `CLAUDE.md` and `README.md`

## 2. Motivation

`/drive` and `/trip` had divergent architectures and produced different artifacts: `/drive` was ticket-driven and reported richly, while `/trip` was instruction-driven and left `/report` thin (its Changes section is built from archived tickets, of which a pure trip had none). Unifying both on the ticket abstraction makes tickets the spine of all work — *sources* (`/ticket`, trip Decomposition) fill the `tickets/todo/` queue, *executors* (`/drive`, context-aware `/trip`) drain it to `tickets/archive/`. This removes special-casing, makes trips reportable and shippable identically to drives, and lets you switch executors mid-queue.

### 3-1. Trip Planning-phase Decomposition gate ([301d389](https://github.com/qmu/workaholic/commit/301d389))

After the design is fixed, the Constructor decomposes `designs/design-vN.md` into implementation tickets under `.workaholic/tickets/todo/<user>/`, each carrying the mandatory `## Policies` (propagated from the design) and a Trip Origin link back to the design section, with `depends_on` from the delivery plan. Emission only.

### 3-2. Trip Coding phase reworked into a per-ticket drive ([13e4f22](https://github.com/qmu/workaholic/commit/13e4f22))

The Coding phase walks the ticket queue in dependency order; per ticket Constructor implements (reading its `## Policies`) → Architect reviews → Planner E2E → archive via `drive/archive.sh`. The three-agent QA is the per-ticket gate, so it stays night-mode-safe.

### 3-3. Report and docs convergence ([e18db4c](https://github.com/qmu/workaholic/commit/e18db4c))

`/report` trip-mode now builds its Changes section from the archived tickets a trip produces and adds a step linking the story to the trip's `trips/<name>/designs/` rationale; CLAUDE.md/README document the ticket-centric trip model.

### 3-4. Context-aware `/trip` (queue-execute mode) ([2cb4596](https://github.com/qmu/workaholic/commit/2cb4596))

`/trip` detects the todo queue: an instruction runs design-first; no instruction + a populated queue runs queue-execute (the team share-reads the queue and trip-drives it with three-agent QA, no design phase); empty + empty tells the user to `/ticket` first. This adds the `ticket → trip` direction.

### 3-5. Document the sources × executors model ([a465a63](https://github.com/qmu/workaholic/commit/a465a63))

CLAUDE.md and README document the unified framing: the ticket is the spine, sources fill the queue, executors drain it, and `/trip` is context-aware — so `/ticket`, `/drive`, and `/trip` read as one system.

## 4. Outcome

`/trip` and `/drive` now share one lifecycle (`todo/ → archive/`) and differ only in *who implements* and *what the per-ticket gate is*:

- The trip Planning phase ends with a Decomposition gate that emits standard tickets (with `## Policies` + Trip Origin) under `tickets/todo/`.
- The trip Coding phase drives that queue per ticket with three-agent QA, archiving each via `archive.sh` so `/report`/`/ship` work identically to a drive.
- `/trip` is context-aware: design-first on an instruction, queue-execute over an existing queue — closing the `ticket → trip` gap.
- `trips/` holds the rationale (why), `tickets/` the contract (what), linked by Trip Origin.

## 5. Historical Analysis

This completes a multi-step convergence. It builds on the recorded `## Policies` ticket section (v1.0.61 on `main`) — the trip design artifact already carried a Policies list described as "the trip's equivalent of a ticket's `## Policies` list," which the Decomposition gate now propagates into emitted tickets. The context-aware entry mirrors the existing pattern already applied to `/report` and `/ship` (detect context, route behavior), giving the command surface a consistent convention. The net effect is that the ticket becomes the sole work abstraction across both commands.

## 7. Successful Development Patterns

- **The ticket as the single spine.** Treating tickets as the one unit both `/drive` and `/trip` produce and consume removed command branching and made trips reportable/shippable with zero special-casing — sources fill the queue, executors drain it.
- **Context-aware routing mirrors `/report` and `/ship`.** Reusing the established "detect state, route behavior" pattern for `/trip` kept the new entry logic consistent and easy to reason about, and used an existing queue-check script (`list-todo.sh`) rather than inline shell.
- **Per-ticket three-agent QA as a self-governing gate.** Architect review + Planner E2E before archiving each ticket is sufficient quality control without a developer prompt, so the trip Coding loop stays night-mode-safe.
- **Decouple rationale from contract, link them.** Keeping the design in `trips/` and each ticket's Trip Origin link to it preserves the "why" one click from the "what" without duplicating it into the ticket.
- **Reuse scripts over parallel paths.** Per-ticket archiving reuses `drive/archive.sh` (not a second archive path), keeping tickets in the standard `archive/<branch>/` location that `/report` already reads.
