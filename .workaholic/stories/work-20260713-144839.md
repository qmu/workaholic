## 1. Overview

This branch makes two long-lived knowledge artifacts — missions and deferred concerns — actionable and self-maintaining. Missions gain an `assignee` and an always-on **mission lens** hook that re-anchors each developer's session on their assigned roadmap. Deferred concerns gain a **stable identity** so they are updated in place instead of re-cloned, a **living migration** that collapses the historical `carried-from` pile (65→32 on this repo), and a **triage step** in `/report` that merges duplicates, combines compound risks, and closes won't-fix items. It also lands the worktree-credentials protocol that copies the root `.env` into new worktrees.

**Highlights:**

1. Missions carry an `assignee`; `mission-lens.sh` surfaces each developer's assigned mission (progress + next criterion) on every `UserPromptSubmit` and `Stop`, strictly gated and non-forcing.
2. Deferred concerns are keyed on a stable `concern_id` and **updated in place** — the `NN-carried-from-…` accumulation is killed at the source.
3. A living, idempotent migration collapses legacy carry chains into one fresh file per concern (most-severe severity, `superseded_by` on the parts).
4. `/report` gains a triage step: judge-proposed A+B compounds, and idempotent `merge-concerns.sh` / `close-concern.sh` mutators the developer drives.

## 2. Motivation

Missions tracked *where work was heading* but tied to no one, and a long session could drift off the roadmap; the fix was an owner on the mission plus a hook that re-injects the mission's status at the turn boundaries, choosing *inform, not force* and *strict ownership*. Deferred concerns had the opposite problem — they never left: a still-active concern was re-cloned into a new file every PR, so 65 files hid ~32 real concerns and old ones only got older. The developer directive was blunt — concerns must be *updated, not appended; always fresh*. That required three moves: a stable identity so extraction updates rather than clones, a migration to collapse the legacy pile, and a human-driven triage for the judgement calls (which near-duplicates are the same concern, when A+B is a compound risk, what is a deliberate won't-fix). Separately, parallel worktrees needed the root `.env` copied in, since `git worktree add` never carries an ignored file.

The branch moves from mission awareness into the concern-lifecycle rework: first the freshness engine (identity + update-in-place + migration), then the triage layer that builds on it. Everything is verified hermetically and the generated `outputs/` tree is rebuilt in lockstep.

### 3-1. ensure-worktree.sh copies the root .env into new worktrees ([8fba48f](https://github.com/qmu/workaholic/commit/8fba48f))

After `git worktree add`, the protocol copies `${repo_root}/.env` into the new worktree (a copy, not a symlink), silently skipped when absent, with the JSON output contract untouched.

### 3-2. Mission assignee and always-on mission lens ([5141637](https://github.com/qmu/workaholic/commit/5141637))

Missions gain an `assignee` (self-assigned to the creator's git `user.email`, surfaced by `list.sh`). `hooks/mission-lens.sh` names each active mission whose `assignee` matches the current `user.email`, with its `checked/total` and next unchecked acceptance item: model-visible `additionalContext` on `UserPromptSubmit`, user-visible `systemMessage` on `Stop`. Strictly gated and non-forcing; a Stop hook cannot inject model context without forcing, so the model half rides `UserPromptSubmit`.

### 3-3. Deferred concerns keyed on a stable identity, updated in place ([b33d64f](https://github.com/qmu/workaholic/commit/b33d64f))

Concerns gain `concern_id`/`first_seen`/`last_seen`. `extract-deferred-concerns.sh` becomes update-or-create: an active concern is refreshed in place (bump `last_seen`, escalate severity, refresh text) — no new file; an archived one is skipped; a new one creates `<concern_id>.md`. A new `migrate-concern-identity.sh` back-fills identity and collapses legacy `carried-from` chains into one file per concern (earliest `first_seen`, latest `last_seen`, most-severe severity, `superseded_by` on the parts), wired living into `extract` and `list-active`.

### 3-4. Concern triage step in /report with compound merges ([da318d7](https://github.com/qmu/workaholic/commit/da318d7))

`/report` Phase 1b fires when the active count exceeds 20 or the judge proposes a compound. The judge's output gains a `compounds[]` proposal (A+B where the combined risk exceeds the parts); the developer confirms/edits via a `[project]`-labelled prompt and applies decisions through idempotent `merge-concerns.sh` (fold members into a compound, escalate severity, archive parts as `superseded_by`) and `close-concern.sh` (archive with `status: resolved|accepted` + reason).

## 4. Outcome

Missions are individually actionable — any session working a worktree is re-anchored to its developer's assigned mission at every turn, silent unless one applies. The concern corpus is self-maintaining: it no longer clones, the legacy pile collapsed on this repo from **65 → 32** active concerns on this very report run (the living migration), and the triage merged the remaining title-drift duplicates. All verification passed: `posix-lint`, `build.mjs`/`verify.mjs`/`validate-metadata.mjs`, and **432/432** smoke tests (+36 new concern assertions), plus hermetic coverage of update-in-place, migration collapse, and the merge/close mutators.

## 6. Concerns

Separately, **32 pre-existing deferred concerns** remain `still_active` — none are remediated by this branch's scoped work; the living migration and triage collapsed them from 65 to 32 and merged the title-drift duplicates, and they persist as fresh identity-keyed files under `.workaholic/concerns/` (they are not re-pasted here to keep this section focused)._

### list-active-deferred-concerns.sh can emit transiently-invalid JSON during the first large migration

- **Severity:** moderate
- **Description:** On the very first run that collapses a large legacy backlog, `list-active` emitted one malformed JSON record (an unescaped control character) while the migration was mutating ~200 files in the same call; subsequent runs were stable and valid (see [da318d7](https://github.com/qmu/workaholic/commit/da318d7) in `plugins/workaholic/skills/report/scripts/list-active-deferred-concerns.sh`). The root cause was not fully isolated. Because the deferred-concern judge parses this output, a first-collapse run in a fresh repo could feed the judge invalid JSON.
- **How to Fix:** Build the entire JSON array in a single Python pass (as `migrate`/`extract` already do) instead of shell string-assembly with per-field `escape_json`, or validate the assembled output and retry once; add a hermetic test that runs `list-active` on a large unmigrated fixture and asserts valid JSON.

### The identity migration under-collapses title-drift near-duplicates

- **Severity:** low
- **Description:** `concern_id` is the slug of the title; when the same concern was re-worded across PRs (e.g. "50-char cap" vs "50-char subject cap"), the migration leaves two active files with different ids, so the mechanical collapse got 65→38 and the remaining ~6 duplicates needed the human triage merge (see [b33d64f](https://github.com/qmu/workaholic/commit/b33d64f) in `plugins/workaholic/skills/report/scripts/migrate-concern-identity.sh`).
- **How to Fix:** This is by design (semantic dedup is the triage's job), but a fuzzy-match hint (shared significant-word overlap) could pre-cluster likely duplicates and offer them as merge candidates in Phase 1b.

### The living migration commits a large one-time churn inside a normal /report commit

- **Severity:** low
- **Description:** The first collapse renamed/moved ~200 concern files; those staged moves ride the Phase 4 story commit, producing a very large, hard-to-review diff bundled with the story (see [b33d64f](https://github.com/qmu/workaholic/commit/b33d64f)).
- **How to Fix:** Consider committing the migration collapse as its own dedicated commit (separate from the story), or gate it behind a one-time `migrate` invocation, so the healing diff is reviewable on its own.

### Triage threshold and compound detection are prose-driven, not enforced

- **Severity:** low
- **Description:** The count threshold (20) and the "judge proposes compounds" step live in report SKILL prose, not a machine check (see [da318d7](https://github.com/qmu/workaholic/commit/da318d7) in `plugins/workaholic/skills/report/SKILL.md`) — matching the repo's existing prose-gate precedent, but skippable.
- **How to Fix:** If enforcement is wanted, have `list-active` emit the active count and a `should_triage` flag the command can branch on mechanically.

## 7. Successful Development Patterns

- Verifying the platform contract before designing (a Stop hook cannot inform the model without forcing) turned an apparent Stop-hook task into the correct `UserPromptSubmit` mechanism.
- Splitting the concern work into a freshness *engine* (identity + update-in-place, independently shippable) and a *triage* layer on top kept each ticket independently testable and let the engine's collapse land before the human-judgement layer.
- Reusing the mission subsystem's proven shapes — stable slug key, living idempotent migration, shared single-writer mutators — meant the concern lifecycle inherited battle-tested patterns instead of inventing new ones.
- Hermetic throwaway-repo tests for every mutator (collapse, escalation, `superseded_by`, idempotency, bad-input rejection) caught behavior before shipping and turned the Quality Gate into a machine check.
