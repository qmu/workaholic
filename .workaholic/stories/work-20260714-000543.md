## 1. Overview

This branch delivers three feature groups. First, a read-only **`summary` mode** for `/ticket`, `/trip`, and `/mission` that reports the current developer's assigned work per command — without changing those commands' existing bare-invocation contracts. Second, the **mission-worktree model**: `/mission "<title>"` now spins up a dedicated, *persistent* `.worktrees/<mission-slug>/` worktree holding the mission statement and ordered kickoff tickets so the developer opens it and immediately `/drive`s; `/ship` preserves that worktree (re-branching from `main`) instead of deleting it; `/mission close` is the only point that tears it down; each worktree gets a unique local port base so concurrent worktrees never collide; missions carry a Playwright-verified quality gate; and the mission lens is scoped to the current worktree's mission. Third, a **release-safety scan** (the new `release-scan` skill) that scans the branch diff for credentials, oversized/too-many files, and unrelated client-work terminology leakage — warning at `/report` and blocking pre-merge at `/ship` — with a committed `.workaholic/scan-allow` false-positive allowlist. Two latent bugs were fixed along the way.

## 2. Motivation

Developers had no on-demand, current-user view of their assigned work per command (`/catch` is by-all-developers and window-based; the mission lens only nudges). Missions had no drive-ready scaffolding or isolation model, so they competed for the main tree and shared `localhost` ports — the developer's confirmed vision is that a mission is a *persistent, isolated* development environment (a dedicated worktree that outlives the branches driven inside it, enabling parallel/overnight development). And release safety leaned on unenforced prose and narrow checks: nothing scanned the branch diff for credentials, bloat, or leaked client terminology before it reached a public remote — the "keep motivation generic, never name other repos/clients" convention lived only in memory and policy prose.

## 3. Changes

### 3-1. Add `summary` mode to /ticket, /trip, /mission ([5ba984a](https://github.com/qmu/workaholic/commit/5ba984a))

Bare `/ticket` or `/ticket summary` lists your assigned todo tickets; `/mission summary` lists your assigned active missions with computed progress; `/trip summary` reports trips plus the todo queue. Bare `/mission` (list) and bare `/trip` (queue-execute) are unchanged. Reuses `list-todo.sh`, `progress.sh`/`next-acceptance.sh`.

### 3-2. Start a topic branch on /mission create ([a9d2631](https://github.com/qmu/workaholic/commit/a9d2631))

`/mission "<title>"` began starting a `work-*` topic branch when on main (later superseded by the worktree model below).

### 3-3. Document the root .env convention in the branching skill ([ea09d21](https://github.com/qmu/workaholic/commit/ea09d21))

Added AI-facing guidance: the root git-ignored `.env` is the single credential source; `ensure-worktree.sh` copies it into new worktrees; a pre-existing worktree needs a manual `cp`.

### 3-4. Mission-keyed persistent worktree primitive ([0abd69a](https://github.com/qmu/workaholic/commit/0abd69a))

`create-mission-worktree.sh` makes `.worktrees/<slug>/` (descriptive dir name) on a fresh `work-*` branch off `main`, copies the root `.env`, and adds `.worktrees/`/`.env` to `.git/info/exclude`; `cleanup-mission-worktree.sh` removes it without discarding uncommitted work; `list-all-worktrees.sh` tags mission worktrees `type: "mission"`.

### 3-5. Create missions in a dedicated worktree ([540249a](https://github.com/qmu/workaholic/commit/540249a))

`/mission` create now derives the slug (`slug.sh`, shared with `create.sh`), spins up the worktree, writes `mission.md` inside it, runs the full `/ticket` flow per kickoff ticket (mission-linked, ordered), and commits — reporting the path without a `cd` instruction (relying on `/drive` auto-routing).

### 3-6. Scope the mission lens to the current worktree ([a5a5f65](https://github.com/qmu/workaholic/commit/a5a5f65))

Inside a mission worktree the lens surfaces only that mission; in the main tree it hides worktree-owned missions and shows only worktree-less ones. The `assignee == git user.email` gate is unchanged.

### 3-7. Reset the mission worktree on ship ([6009bb3](https://github.com/qmu/workaholic/commit/6009bb3))

`/ship` preserves a mission worktree on merge and resets it (fresh `work-*` branch off `main`) instead of deleting it; ordinary drive/trip worktrees still get cleaned up.

### 3-8. Tear down the mission worktree on close ([8757d8c](https://github.com/qmu/workaholic/commit/8757d8c))

`/mission close` removes the mission's worktree after archiving the mission — never discarding uncommitted work (refuses a dirty worktree, still closes the mission).

### 3-9. Assign a unique port base per mission worktree ([ba147e8](https://github.com/qmu/workaholic/commit/ba147e8))

Each mission worktree gets a collision-free `WORKAHOLIC_PORT_BASE` (`4100 + k*10`) in its `.env`, so several worktrees run dev/docs servers at once without clashing (and each can be Playwright-driven independently). Freed bases reuse.

### 3-10. Add a per-mission quality gate ([efef757](https://github.com/qmu/workaholic/commit/efef757))

Missions declare `gate_type` (`documentation`/`live-app`), `gate_target`, `gate_assert`; `gate.sh` reads it and resolves the worktree's port; `/drive` surfaces a missioned ticket's gate and verifies it via Playwright against that port (server-start stays project-owned).

### 3-11. Add the branch-safety scan engine ([c5f4b9c](https://github.com/qmu/workaholic/commit/c5f4b9c))

`release-scan/scan-branch-safety.sh` scans `git diff <base>..HEAD` → `{verdict, findings[]}` for `secret` (hard), `size` (override), `leak` (confirm), each citing `file:line` + rule with secret evidence redacted. Leak detection uses a git-ignored `.workaholic/leak-denylist` + internal-hostname patterns. A script, not a subagent, so a merge gate stays objective.

### 3-12. Gate report and ship on the scan ([0fb0888](https://github.com/qmu/workaholic/commit/0fb0888))

`/report` runs the scan as a loud warn (forces `releasable: false` on a hard/confirm block); `/ship` runs it pre-merge — `secret` non-overridable, `size`/`leak` block with a recorded accepted-risk override. `gate-decision.sh` makes the tier policy deterministic.

### 3-13. Add a false-positive allowlist to release-scan ([b68f95f](https://github.com/qmu/workaholic/commit/b68f95f))

Dogfooding surfaced that the scan hard-blocked this very branch on its own pattern docs and test fixtures. `scan-branch-safety.sh` now honors a committed, reviewed `.workaholic/scan-allow` (git pathspec globs) excluding known-safe paths; secrets stay non-overridable at ship time.

## 4. Outcome

**Changed:** `/mission` create (topic branch → dedicated worktree); the root-`.env` guidance; `/ship` worktree preservation; `/mission close` teardown; mission-lens worktree focus; per-worktree ports; `/report`+`/ship` scan wiring.

All 12 tickets completed as planned. The full hermetic suite grew to **524 passed / 0 failed**; `build.mjs`/`verify.mjs`/`validate-metadata.mjs`/`posix-lint` all clean.

## 5. Historical Analysis

- **Reuse over reinvention:** shared primitives (`slug.sh`, `gate-decision.sh`, the factored secret-pattern lib) enforce single-source-of-truth; composing `check.sh → create.sh` beats one-shot utilities.
- **Hermetic tests earn their keep:** throwaway-repo tests (no network) validated worktree lifecycle, lens scoping, ports, and the scan — and surfaced two latent bugs that had shipped untested.
- **Cross-skill script reuse pulls whole build closures:** referencing `drive`/`release-scan` from built skills bundles their scripts into `outputs/`; expect larger `outputs/` diffs and rebuild when scripts are shared.
- **Dispatch order matters:** a `summary`/keyword mode must be routed before a command's content-consuming branch, or the keyword is swallowed as a title.

## 6. Concerns

### Branch-safety scan WARN (overridable): oversized change set

- **Severity:** low
- **Description:** The scan's only remaining finding on this branch is `size/too-many-files` (103 > 100) — legitimate, because `outputs/` was regenerated across many commits. It is `override`-tier and is recorded at `/ship`, not a block.
- **How to Fix:** None required; `/ship` records the override in deployment evidence.

### ensure-worktree.sh lacks the `.git/info/exclude` embedding guard

- **Severity:** moderate
- **Description:** The mission-worktree path adds `.worktrees/`/`.env` to `.git/info/exclude` to stop `git add -A` from embedding a linked worktree as a gitlink, but the pre-existing `ensure-worktree.sh` (trip/drive worktrees) still lacks the same guard.
- **How to Fix:** Apply the same `.git/info/exclude` guard in `ensure-worktree.sh` (a small parity follow-up).

### report step-1 wording predates the tiered scan severities

- **Severity:** low
- **Description:** `report/SKILL.md` step 1 says "force `releasable: false` on a block," but the scan now has an `override` tier that should warn-and-record rather than block. The release-readiness pass handled this correctly (it treated the override size finding as non-blocking), but the wording should distinguish `hard`/`confirm` (force not-releasable) from `override` (warn).
- **How to Fix:** Reconcile the step-1 wording to key off severity, not just "block."

### Mission quality gate: server-start and live verification are not hermetic

- **Severity:** low
- **Description:** The per-mission gate declares the check and supplies the worktree port, but the server-start command is project-owned and the live Playwright verification is an in-session step, not covered by the hermetic suite (the schema round-trip is).
- **How to Fix:** If the run-side grows, split the declaration from the verification-run at drive time.

## 7. Successful Development Patterns

- Extracting shared logic to canonical scripts (`slug.sh`, `gate-decision.sh`, `secret-patterns.sh`) to enforce single-source-of-truth.
- Composing scripts from simple primitives rather than complex one-shot utilities.
- Hermetic smoke tests that catch bugs without touching the working tree — and that turned dogfooding into a real defect find (the scan blocking its own branch).
- Per-ticket verification gates (test + build + verify + posix-lint) before each archive, keeping the branch continuously green.
- Making a policy objective and testable (the ship tier logic as `gate-decision.sh`) rather than leaving it to agent interpretation.
