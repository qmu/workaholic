# Agent-orchestrated development

## Direction

Move workaholic's own development onto an after-hours agent-orchestration model, and shape the workflows so any team can do the same.

The working day is spent on **planning**, not typing: the developer interrogates each mission to question-free, drive-ready readiness, front-loading every judgment call an executing agent would otherwise have to ask. After hours, a fleet of coding agents executes those missions in parallel — long-running, unattended, each in its own worktree — so the morning begins with reviewable results (open PRs, honest completion reports, recorded reflections) rather than a blank editor.

The bet behind this is an economic one: a coding agent's mistake is cheaply corrected by a later agent — code keeps getting cheaper — while the scarce resource is developer attention. So the direction is to spend attention up front on planning and review, and to spend cheap agent-time lavishly overnight; to keep every question the system asks a genuine one; and to make the orchestration itself *measurable* — how many agents, kept running for how long, producing how much reviewable change — so the team can tell whether it is orchestrating its fleet well.

This is direction, not a finish line: there is no state in which "agent-orchestrated development" is *done*. Missions are the execution plans that move workaholic along it — the mission/strategy split, duration prediction, overnight reflection, the throughput KPI, and the granularity discipline are all missions launched to execute this strategy, and more will follow.
