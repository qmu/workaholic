# Record the standard policies a ticket answers to, and have /drive and /trip refer to them

## Overview

Make it explicitly confirmable that, when this plugin is used, implementation refers to the standard engineering policies synced from the corporate site (qmu.co.jp). Add a mandatory `## Policies` section to every implementation ticket that lists those policies as a durable, in-ticket record. A Claude Code session that **drives** over the ticket reads that recorded list and opens each named policy before writing code; a session that **trips** does the equivalent through the Constructor's design artifact.

Today the policy lens is preloaded only while a ticket is *written* (`/ticket`'s `policy-lens.sh` hook), but `/drive` and `/trip` implement *later*. Without a recorded list in the ticket itself, there is nothing a developer can inspect after the fact to confirm which standard policies the implementation answered to. This ticket closes that gap.

## Policies

The standard engineering policies — synced from the corporate site (qmu.co.jp) into the `workaholic` policy skills — that govern this ticket. The implementing session **MUST** read each linked policy hard copy before writing code and keep every change defensible against that policy's Goal (目標), Responsibility (責務), and Practices (実践).

- `workaholic:implementation` / `policies/directory-structure.md` — conventional project layout: author changes under `plugins/` only, never `.claude/`; regenerate the committed `outputs/` mirrors rather than hand-editing them.
- `workaholic:implementation` / `policies/coding-standards.md` — style/convention conformance for all source-tree edits (applies to all code/config work).
- `workaholic:implementation` / `policies/objective-documentation.md` — the added prose must factually describe what `/drive` and `/trip` actually do (read the section, open the listed policies), not overstate enforcement; documentation is part of the codebase.
