# Diagram Generation from Code

_Prefer diagrams that can be generated or updated from the code or configuration they describe; a diagram that is maintained separately from its subject drifts._

Architecture diagrams, data flow charts, entity-relationship diagrams, and sequence diagrams have real value — they make implicit structure explicit and allow reasoning about a system at a level above individual files. But diagrams maintained by hand in design tools tend to go stale: they are updated when someone thinks to update them, not when the code changes. We prefer diagrams expressed in text-based formats (Mermaid, D2, PlantUML) embedded in documentation or generated from the code and schema, so that the diagram and its subject can be reviewed in the same PR.

## Goal (目標)

The situation this policy aims to achieve is one in which diagrams that describe the system are current, because they are embedded in or generated from the files they describe, and are updated in the same PR as the thing they document.

- Architecture and data-flow diagrams for significant systems are present in the repository in a text-based format.
- ER diagrams for the database schema are generated from the schema definition file, not maintained by hand.
- Diagrams that are no longer accurate are detected because they live next to the code that falsifies them.

## Responsibility (責務)

The situation this policy aims to prevent is one in which the diagrams that should describe the system are either absent or out of date — providing false confidence rather than accurate documentation.

States we do not tolerate:

- Architecture diagrams maintained only in a design tool (Figma, draw.io) that is not version-controlled alongside the code.
- ER diagrams that were last updated when the schema was first written and have not tracked migrations.
- Diagrams referenced in documentation that cannot be opened and regenerated by a developer who does not have access to the original tool.

## Practices (実践)

### Use Mermaid for in-documentation diagrams

For diagrams embedded in Markdown documentation (README, ADR, wiki), use Mermaid. GitHub renders Mermaid natively in Markdown. Mermaid's text-based format means the diagram is reviewed in the same PR as the text it accompanies.

### Generate ER diagrams from the schema

Use a tool that generates an ER diagram from the database schema automatically (e.g., `prisma-erd-generator` for Prisma, `tbls` for raw SQL, `dbdocs` for DBML). Run generation in CI and commit the output to the repository. The generated diagram is always current as of the last migration.

### Limit diagram scope to what a reader needs in one reading

A diagram that tries to show the entire system at once becomes unreadable. Define a diagram's scope by audience and question: "the data flow for the payment processing path" rather than "the full system architecture." Maintain multiple smaller, focused diagrams rather than one comprehensive one.
