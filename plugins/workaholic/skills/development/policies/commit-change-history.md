# Preserving Change History in Files

_Storing tickets, stories, and release notes as markdown files in the repository alongside code, so that the rationale behind each change is retrievable from a single clone._

In our development, in addition to code changes themselves, we store the intent and background behind those changes — the change history — as markdown files in the repository. Tickets, branch-level stories, and user-facing release notes are placed in directories within the repository, included in the same commit as the code changes, and pushed together. The location is files in the repository, not an issue tracker, chat system, or internal wiki.

Code, commit history, change history, and decision records are things we want to keep together rather than separate. Keeping them in the same repository means that a `git clone` is enough to have not only code but also history in a consistent state.

In a firm where AI writes most of the implementation (see [Active Use of Generative AI](ai-utilization.md)), design judgment criteria are carried by plugins ([Policies Distributed as Plugins](policy-as-plugin.md)) into the model's context. What plugins carry is up to organization-level design standards. "What happened in this project in the past" and "why did this change take this shape" — project-specific history — is placed not in organization-distributed materials but in the project's repository itself.

## Goal (目標)

We aim for a state in which code, commit history, change rationale, and decision records all reside in the same repository, reachable with a `git clone` and `git log`. We aim for a state in which the next time AI touches the same code, `grep`, `git log`, and `.workaholic/tickets/archive/` are all in the same working set, with code, history, and rationale available from a single clone. This extends the principle of making code self-explanatory through naming and structure (see [Codifying Domain Terminology](../../planning/policies/terminology.md)) to history as well — not just code, but why the code took that shape, is traceable from the same place.

## Responsibility (責務)

Our responsibility is to prevent a state in which change rationale and decision records live only in SaaS-side issue trackers or chat systems from which they cannot later be recovered into the repository. Once history and decisions accumulate on the SaaS side, extracting them back into the repository has limited means available today. To avoid that, we decide at project start that all history and decisions go into the repository.

## Practices (実践)

### Document files: tickets, stories, and release notes in the repository

Materials larger than a single commit — tickets, branch-level stories, user-facing release notes — are placed as markdown in directories within the repository. They are not placed in GitHub Issues, PR comments, Discussions, Jira, Pivotal Tracker, Redmine, or other project management SaaS. Epic management continues to use external project management services; the granularity treated as epics in our operation has grown larger compared to before LLM coding agents. Individual tickets and decision records take the repository file form.

### Do not micro-manage individual tickets

The practice of queuing development work and prioritizing it was a method established in the era when individuals implemented each task. With LLM coding agents writing much of the implementation, circumstances have changed: a single task's implementation time has shortened, and the overhead of assignment, priority-update, and status-transition ceremonies can exceed the work itself for fine-grained changes. Therefore, individual development issues are not micro-managed — they proceed at the developer's discretion. The developer decides together with the coding agent what to create and in what order. When a branch is merged and the tickets in `todo/` are gone, that work is complete.

### Commit messages generated by AI coding agent

Commit messages are not written by hand but generated by AI coding agents. The agent is given the relevant ticket, the actual diff, and related implementation policies as context, and writes the change's motivation, changes made, verification steps, and release readiness as structured text.

Not adopting handwriting is not to "get it done fast" but to avoid quality degradation that varies with the moment's capacity or mood: title-only with empty body, predicate-only like `fix` or `update`, motivation missing — in a handwriting operation, this kind of degradation tends to accumulate over time.

### Commit messages are not treated as central records like document files

Commit messages are generated and preserved this way, but they are not treated with equal or greater weight than document files, for the following reasons:

- **AI reads back and weaves into documents**: The same AI coding agent reads back `git log` as material when writing stories and release notes. Commit message content is woven into document files by the time the branch completes. Commit messages are fine-grained raw material; document files are collected history — layered by granularity.
- **Keeping the option to discard history**: Unintended files — for example, screenshots containing personal information — can make their way into commits. Since partially rewriting pushed old commit history can be difficult, as a last resort there is the judgment of rebuilding the repository and moving only the current files to a new repository. Keeping the center of history in document files means that even if commit history is lost, history can be carried over to the new repository.
- **History is not groomed for appearance**: AI coding agents commit incrementally as work progresses, so reversals where a feature is removed and re-added can remain right up until PR creation. We do not adopt a practice of linearizing with squash or rebase. With history in place on the document file side, leaving the trial and error of that time tends to be more useful to later workers.
