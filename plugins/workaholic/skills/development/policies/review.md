# Code Review as a Non-Default Practice

_Removing code review from the default development workflow in favor of policy-grounded generation and each developer's own QA, with mutual awareness maintained through generated stories and commit logs._

We remove code review — the practice of humans reading through AI-generated code together after the fact — from the default development procedure. Code review has functioned broadly as a quality support measure in a setup where humans write most of the implementation. We remove it from the default not because it has no value but because we are changing the nature of development itself: as AI writes more of the implementation, we move to a setup in which developers proceed without reading every line of code.

The quality of generated code is largely determined far upstream from the moment of generation. Our policies, delivered to AI through Workaholic, shape the generated code itself. Individually correcting already-generated code downstream has no end, and if there is a behavior we want to correct, the right move is to return to the policy side that produced the code. Given that developers proceed without reading the code themselves, it does not follow to ask other developers to review code that one has not read — this leads to the conclusion that review is not conducted as a default.

## Goal (目標)

We aim for a state in which code quality is maintained not by a process of reading back generated code and correcting it after the fact, but by policies before generation and by each developer's own QA. We aim for a state in which, even without developers reading each other's code, the stories and commit logs that record generates, and dialogue with AI, allow each person to grasp what development the others are advancing — the state we are headed toward as a team.

## Responsibility (責務)

Our responsibility is to prevent a state in which removing review from the default leaves the responsibility for quality floating in the air, with no one taking on the correctness of generated code. Removing review is not license to hand quality off to a later process. If each developer does not take on QA as their own responsibility and the policies that produced the code remain uncorrected, quality with no owner piles up. In a setup where generative AI produces large volumes of code, it is easy for code that no one is watching to flow forward on the strength of the fact that it is working and CI is passing.

## Practices (実践)

### Correct code by returning to the policy side

Rather than correcting the branches of already-generated code downstream in daily development, we correct the policy side — the policies on this site and the standards delivered to AI through Workaholic — that produced that code. When there is a behavior we want to correct, we trace back first to which policy produced that output. Individual corrections downstream have no end and will reproduce the same divergence unless the source is corrected.

### Each developer takes on QA

Rather than delegating quality assurance entirely to a dedicated person or a later process, each developer plans, executes, and corrects QA. As the time spent writing code by hand shortens, the QA aptitude of developers — the ability to identify what to verify and verify it — is increasingly demanded. We develop with the awareness of supporting the quality of what we put into the world ourselves, rather than relying entirely on code review. See [Implementers Own Quality Assurance](qa-engineering.md) for details.

### Mutual awareness of development through record's stories and commit logs

Developer-to-developer catch-up starts not from reading each other's code but from the stories generated by Workaholic's record command and from commit logs. Stories are written with the expectation of being read by humans; what other developers have advanced and how is grasped using these as a foundation through dialogue with AI.
